Log Handling Simulation System - Amrita Kumari

Description

This system simulates the generation, parsing, and storage of Nginx logs, providing a complete pipeline from log generation to storage in a database with accessible APIs for querying.

Components

  • Log Generator:

    • Generates logs in the Nginx log format at a rate defined by the environment variable RATE (e.g., 100/s, 2000/h, 300/m).
    • Sends the generated logs to the Log Parser Server via HTTP calls.

  • Log Parser Server:

    • Accepts logs and converts them into JSON format. and print in log file at log parser server side.
    • Stores parsed logs in a Postgres database. Database design is flexible.
    • Provides APIs to read logs from the database with filters such as timestamp, IP, status_code, and body_bytes_sent. Combination of filters is supported.
    • Offers basic statistics based on filters, such as count of logs.

  • Postgres:

    • Stores the log information.

Deliverables

  • A Dockerfile and unit tests for each component.
  • The system must handle edge cases efficiently.
  • Developers can incorporate additional functionality as deemed useful.
  • A Helm chart for Kubernetes deployment and a demonstration.

HTTP Traffic Simulation System - Jathin S

Description

This system simulates HTTP traffic generation and collection, storing the data in a database and offering querying capabilities.

Components

  • Traffic Generator:

    • Generates HTTP traffic based on environment variables:
      • NO_OF_API: Number of APIs to simulate (e.g., 10, 100).
      • API_RATE: Rate of API calls (e.g., 100/s, 2000/h, 300/m).
    • Maximum number of APIs is 2^13.
    • Sends traffic data to the Traffic Stats Collector.

  • Traffic Stats Collector:

    • Collects statistics from incoming API data.
    • Uses Postgres to store statistics. Database design is flexible.
    • Provides a read API with filters on METHOD, API, STATUS, BYTE_SIZE, or any combination.
    • Offers statistics such as total count, average byte size, and unique domains, with timestamp filtering.

  • Postgres:

    • Stores traffic information.

Deliverables

  • A Dockerfile and unit tests for each component.
  • The system must handle edge cases efficiently.
  • Developers can incorporate additional functionality as deemed useful.
  • A Helm chart for Kubernetes deployment and a demonstration.

Log Files Generation Simulation - Pintaram M

Description

This system generates log files, transfers them to an SFTP server, parses them, and stores the parsed data in a database.

Components

  • SFTP Server:

    • Stores generated log files.

  • Log File Generator:

    • Generates log files based on environment variables:
      • NO_OF_FILES: Number of files.
      • FILE_SIZE: Size of each file (e.g., 10MB, 1GB, 5KB).
      • FILE_RATE: Rate of file generation (e.g., 100/s, 2000/h, 300/m).
    • Transfers files to the SFTP server.

  • Log File Parser:

    • Periodically collects logs from the SFTP server based on POLLING_TIME (e.g., 1s, 1m, 1h).
    • Parses logs and stores them in Postgres.
    • Provides a read API with filters on METHOD, API, STATUS, BYTE_SIZE, and any combinations.
    • Offers statistics such as total count, average byte size, and unique domains, with timestamp filtering.

  • Postgres:

    • Stores parsed log information.

Deliverables

  • A Dockerfile and unit tests for each component.
  • The system must handle edge cases efficiently.
  • Developers can incorporate additional functionality as deemed useful.
  • A Helm chart for Kubernetes deployment and a demonstration.

System Metrics Collection and Display System - Anil Kumar

Description

This system involves collecting system metrics (CPU, RAM, disk, and process information) and providing an API to access these statistics.

Components

  • Metrics Agent:

    • Collects system metrics including CPU usage, RAM usage, disk space, and running process information.
    • Sends the collected metrics to the Metrics Collector at a regular interval defined by an environment variable COLLECTION_INTERVAL (e.g., 10s, 1m, 5m).

  • Metrics Collector:

    • Receives metrics data from the Metrics Agent.
    • Stores the metrics in a Postgres database. Database design is flexible.
    • Provides an API to access the stored metrics with filters such as time range and specific metric types (e.g., CPU, RAM).
    • Offers basic statistics such as average CPU usage, total RAM usage, and disk space over a specified period.
    • API output is upto the developers but it should be able to provide below metrics at least ex,
